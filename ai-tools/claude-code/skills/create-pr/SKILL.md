# create-pr

Open a GitHub pull request from the current feature branch into the repo's default branch.

## Workflow

### 1. Branch guard — BLOCKING, runs first

Before anything else, determine the current branch:

```bash
git rev-parse --abbrev-ref HEAD
```

**Refuse to proceed** if the result is any of:

- `main`
- `master`
- `HEAD` (detached HEAD state)
- empty / error

Tell the user a PR cannot be opened from this branch and that they need to create a feature branch first. Do **not** call `gh` or push anything. Stop here.

There is no override flag for this check.

### 2. Inspect repo state

Run in parallel:

- `git status` (no `-uall`) — uncommitted changes
- `git rev-parse --abbrev-ref --symbolic-full-name @{u} 2>/dev/null` — upstream tracking
- `gh repo view --json defaultBranchRef -q .defaultBranchRef.name` — base branch (fall back to `main` if `gh` errors)
- `gh pr view --json number,url,state 2>/dev/null` — does this branch already have an open PR?

If `gh pr view` returns an open PR, stop and report the existing URL — do **not** create a duplicate.

Then, with `<base>` resolved:

- `git log <base>..HEAD --oneline` — commits the PR will include
- `git diff <base>...HEAD` — full diff to draft the body from

### 3. Pre-flight

- If `git status` shows uncommitted work that belongs in this PR, stop and tell the user to commit first (suggest `/stage-commit-push`). Do not stage on their behalf here.
- If the branch has no upstream, or is ahead of its upstream: `git push -u origin <branch>` before creating the PR.
- Never force-push.

### 4. Draft title and body

- **Title**: < 70 characters, imperative mood, summarizes the *full set* of commits in the range — not just the most recent commit. Match the repo's existing commit/PR style (check `git log` and recent PRs).
- **Body**: pass via HEREDOC. Required sections:

```markdown
## Summary
- <1–3 bullets covering all commits in <base>..HEAD>

## Test plan
- [ ] <how to verify>

🤖 Generated with [Claude Code](https://claude.com/claude-code)
```

### 5. Create the PR

```bash
gh pr create \
  --base <base> \
  --head "$(git rev-parse --abbrev-ref HEAD)" \
  --title "<title>" \
  --body "$(cat <<'EOF'
<body>
EOF
)"
```

- Do **not** pass `--draft` unless the user explicitly asked for a draft.
- Do **not** use `--fill` — we want an authored summary, not autogenerated.

### 6. Report

Two lines max:

- PR URL
- Title

## Guardrails

- Refuse on `main`, `master`, or detached HEAD. Non-negotiable. No override.
- Never force-push, never `--no-verify`, never modify git or `gh` config.
- Only run when the user invokes the skill explicitly.
- If `gh` is not installed or not authenticated (`gh auth status` fails), stop and tell the user — do not attempt to authenticate on their behalf.
- Do not push to `main`/`master` under any circumstance.
